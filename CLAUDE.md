## What This Repo Does

`sunbird-spark-installer` automates deployment of the full Sunbird-ED learning platform on Kubernetes (Azure AKS or GCP GKE). It combines OpenTofu (Terraform-compatible) infrastructure provisioning with Helm chart deployments across 7 modular "building blocks".

## Key Commands

All deployments run from `opentofu/<provider>/<env-name>/` after copying the template:

```bash
cp -r opentofu/azure/template opentofu/azure/<env-name>
cd opentofu/azure/<env-name>

./install.sh                              # Full install (all phases in order)
./install.sh create_tf_backend            # Init OpenTofu state backend
./install.sh create_tf_resources          # Provision cloud infra (AKS/GKE, storage, etc.)
./install.sh install_helm_components      # Deploy all 7 Helm bundles
./install.sh install_component <name>     # Deploy a single bundle (e.g., edbb, learnbb)
./install.sh certificate_config           # Inject RSA cert keys into registry service
./install.sh dns_mapping                  # Wait for DNS propagation (20-min timeout)
./install.sh generate_postman_env         # Build env.json for Postman tests
./install.sh run_post_install             # Run Postman collection for API validation
./install.sh create_client_forms          # Seed Sunbird form configurations via Postman
./install.sh destroy_tf_resources         # Tear down all cloud infra (cleans cert keys first)
```

Multiple functions can be chained: `./install.sh func1 func2` — the `invoke_functions` fallback at the bottom of `install.sh` invokes each argument as a function name in order.

For addons (must export `ENV_NAME` first):
```bash
export ENV_NAME=<env-name>
cd addons/<addon-name>/script
./addon.sh install [azure|gcp]
./addon.sh uninstall [azure|gcp]
```

For building/pushing Dockerized scripts (multi-arch `linux/amd64,linux/arm64`):
```bash
# Triggered automatically via .github/workflows/build-push-images.yml
# or run docker buildx build --platform linux/amd64,linux/arm64 from scripts/<dir>/
```

## Architecture

### Building Blocks (Helm Umbrella Charts)

Installation order matters — each bundle builds on the previous:

| Order | Bundle | Key Contents |
|---|---|---|
| 1 | `monitoring` | Prometheus, Grafana, Loki, Grafana Alloy |
| 2 | `edbb` | Kong API Gateway, NGINX ingress, Kafka, Player portal, knowledge-mw |
| 3 | `learnbb` | Lern service, Keycloak, YugabyteDB, Elasticsearch, Redis, Flink jobs, cert services |
| 4 | `knowledgebb` | Knowlg service, Search, JanusGraph (on YugabyteDB), Flink jobs |
| 5 | `obsrvbb` | Telemetry service, Superset dashboards |
| 6 | `inquirybb` | Inquiry service |
| 7 | `additional` | cert-ng, NL web, volume autoscaler |

Each bundle is an umbrella Helm chart in `helmcharts/<bundle>/` with subcharts in `helmcharts/<bundle>/charts/`. All `helm upgrade --install` calls use `--timeout 30m --debug`.

### Helm Value Layering

Every `helm upgrade --install` merges these files in order (later overrides earlier):
1. `helmcharts/<bundle>/values.yaml` — defaults
2. `helmcharts/<bundle>/ed-values.yaml` — education-specific overrides (if present; also adds `--wait --wait-for-jobs`)
3. `helmcharts/images.yaml` — all Docker image refs
4. `helmcharts/global-resources.yaml` — CPU/memory/storage specs for all services
5. `opentofu/<provider>/<env>/global-values.yaml` — environment config (edited by operator)
6. `opentofu/<provider>/<env>/global-cloud-values.yaml` — generated by OpenTofu post-provisioning

When `deployed_dial_addon: true` is set in `global-values.yaml`, `addons/global-cloud-values.yaml` is inserted before step 5 above.

### Subchart Conditions

Subcharts in `Chart.yaml` have `condition: <chart>.enabled` flags. To redeploy a single subchart without running the full bundle, set `--set <target>.enabled=true` and all others to `false` in a manual `helm upgrade --install` call. There is no `install_service` shell function — this must be done manually.

### Namespace Management

- `sunbird` and `velero` — created by the `workload-identity` Terraform module, not by `install.sh`
- `volume-autoscaler` and `nlweb` — created by `install_component` on every run (`kubectl create namespace … || true`)

### RSA Certificate Key Lifecycle

During `learnbb` install, `install_component` calls `certificate_keys()` which generates `certkey.pem` / `certpubkey.pem` and appends 4 keys (`CERTIFICATE_PRIVATE_KEY`, `CERTIFICATE_PUBLIC_KEY`, `CERTIFICATESIGN_PRIVATE_KEY`, `CERTIFICATESIGN_PUBLIC_KEY`) to `global-values.yaml`. These keys are idempotent — generation is skipped if the keys are already present. `destroy_tf_resources` calls `cleanworkspace()` which removes the `.pem` files and strips those 4 lines from `global-values.yaml`.

### Infrastructure (OpenTofu)

- Azure: `opentofu/azure/modules/` — AKS, VNet, Storage, Key Vault, Managed Identity, Workload Identity
- GCP: `opentofu/gcp/modules/` — equivalent GKE modules (GKE, Keys, Network, Storage, Service Account)
- Terragrunt orchestrates module dependency order
- State backend initialized separately via `create_tf_backend.sh` before first apply
- `tf.sh` (sourced inside `create_tf_resources` and `destroy_tf_resources`) sets cloud credentials and Terraform variables; it is auto-generated after backend creation and should not be committed

### Key Databases

- **YugabyteDB**: Primary DB for all building blocks — CQL (9042) and YSQL/PostgreSQL (5433). 6 pods (3 masters + 3 tservers), 120 Gi disk total
- **Elasticsearch**: Full-text search — port 9200
- **JanusGraph**: Graph DB backed by YugabyteDB CQL, used by KnowledgeBB — port 8182
- **Kafka**: KRaft mode (no ZooKeeper), 3 controller pods — port 9092

### Scripts (Dockerized)

`scripts/` contains Dockerized utilities that run as Kubernetes Jobs:
- `kong-api-scripts/` — Kong route/plugin configuration (`kong_apis.py`, `common.py`)
- `sunbird-yugabyte-migrations/` — DB schema migrations
- `keycloak-21.1.2/` — Keycloak realm/client setup
- `janusgraph/` — JanusGraph index initialization

Built and pushed via `.github/workflows/build-push-images.yml`; referenced in `helmcharts/images.yaml`.

## Configuration

`global-values.yaml` (the main operator-edited file) contains:
- Cloud provider credentials and region
- Domain name and SSL certificates
- Google OAuth, ReCAPTCHA keys
- Email SMTP settings
- Feature flags: `deployed_dial_addon`, `enable_asset_enrichment`

The template file contains placeholder values (`REPLACE_WITH_*`). Never commit actual credentials.

Addon endpoint overrides live in `addons/global-values.yaml` (elasticsearch, redis, yugabyte, kafka, janusgraph hosts).

## Deployment Paths

Two patterns for production use (see `private-repo-setup/README.md`):

1. **GitHub Actions CI/CD** — A separate private repo holds encrypted `global-values.yaml` and workflow files. The workflow clones `sunbird-spark-installer` at runtime and invokes `install.sh`. Requires Azure OIDC auth.
2. **Manual via Azure VM** — SSH into an Azure VM and run `install.sh` directly.

## Addons

Optional extensions under `addons/`:
- `dial` — QR code content linking; also provisions cloud storage via OpenTofu modules
- `discussion-forum` — NodeBB community threads; deploys services in dependency order (discussion-forum-apis → discussion-forum-consumers → discussionmw → nodebb → groups)
- `video-stream-generator` — HLS transcoding via a Flink job
- `asset-enrichment` — video/image enrichment Flink job (was previously disabled inside knowledgebb)
- `obsrv` — observability extensions

Each has its own Helm chart and `addon.sh` install/uninstall script. Addons share `addons/global-values.yaml` and `addons/images.yaml`.
